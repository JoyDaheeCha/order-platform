# ADR-0002: 재고 차감 동시성 기법

- **상태(Status)**: Accepted (2026-06-27)
- **관련 정책**: [policy.md](../policy.md) §6(PV-1~5, 특히 **PV-4 오버셀 금지**), §3(PI-5 멱등성), §9(PT-2·3 재시도)
- **위치**: `inventory.infrastructure` 영속화 어댑터 (architecture.md §8)
- **후속 의존**: [ADR-0004](./0004-schema-separation-outbox-readmodel.md) 영속화 구성(컨텍스트별 DataSource) 위에서 구현

---

## 1. 맥락 (Context)

재고 차감은 `PaymentCompleted` 수신 후(`PAID` 단계, PV-2) 일어난다. **동일 상품을 여러 구매자가 동시에 주문**하면 고전적 read-modify-write 경쟁 조건이 발생한다(E5):

```
재고 1개 남음. 구매자 A·B가 동시에 차감 시도.
A: SELECT qty=1 →  B: SELECT qty=1  →  A: UPDATE qty=0  →  B: UPDATE qty=0
결과: 2개 팔림. 물리 재고 1개. ❌ 오버셀
```

PV-4는 이 오버셀을 **불변식으로 금지**한다. 이 ADR은 *그 불변식을 무엇으로 보장할지* 기법을 확정한다.

> **학습 맥락(todo.md)**: "재고 동시성 제어"는 명시적 학습 목표다. 이 프로젝트는 단일 기법 선정보다 **여러 기법을 구현·비교**하는 데 학습 가치가 있다(ADR-0001·0005의 비교학습 패턴과 동일). 따라서 이 ADR은 "하나를 고른다"가 아니라 **"포트로 추상화해 여러 기법을 갈아끼우며 비교한다"**로 결정한다.

제약:
- DB는 **단일 MySQL 인스턴스**(컨텍스트별 스키마, ADR-0004). 재고는 `inventory_schema.stocks`.
- 차감은 주문 단위 **all-or-nothing**(PV-3): 라인 하나라도 부족하면 전체 차감 없이 `StockShortage`.
- 보상 복구(`RestoreStock`)는 정확히 원복·멱등(PV-5).

## 2. 결정 기준 (Decision Drivers)

| # | 기준 | 가중치 |
|---|------|--------|
| D1 | **정합성** — PV-4 오버셀을 확실히 0으로 만드는가 | 높음 |
| D2 | **학습 가치** — 동시성 제어의 서로 다른 철학을 명시적으로 비교 체득하는가 | 높음 |
| D3 | **단순성 / 운영** — 데드락·이중 소스 정합성 등 부수 위험이 적은가 | 중간 |
| D4 | **처리량** — 동시 차감 경합에서 성능 저하가 적은가 | 낮음(학습 규모, 단일 인스턴스) |

## 3. 고려한 옵션 (Considered Options)

네 기법은 "오버셀을 막는 메커니즘"이 근본적으로 다르다 — 이 차이가 학습의 핵심이다.

### 옵션 A — 비관적 락 (`SELECT … FOR UPDATE`)
차감 전 재고 행을 잠가 트랜잭션을 직렬화한다. JPA `@Lock(PESSIMISTIC_WRITE)`.
- 철학: **"충돌이 날 거라 가정 → 미리 잠그고 진행"** (블로킹).
- 장점: 직관적, 강한 정합성(D1 ◎). "락으로 임계영역 보호" 기본기(D2 ◎).
- 단점: 락 경합 시 처리량↓(D4). 멀티라인은 `product_id` **정렬 잠금**으로 데드락 회피 필요(D3 △).

### 옵션 D — 낙관적 락 (`@Version` + 재시도)
버전 컬럼으로 충돌을 *감지*하고 재시도한다.
```sql
UPDATE stocks SET qty=qty-:n, version=version+1 WHERE product_id=:id AND version=:read;
-- affected rows=0 → 그새 변경됨 → OptimisticLockException → 재시도
```
- 철학: **"충돌은 드물 거라 가정 → 일단 진행, 깨지면 재시도"** (논블로킹).
- 장점: 락 미점유로 경합 낮을 때 빠름(D4). 비관 락과의 **대조**가 학습 핵심(D2 ◎).
- 단점: **재고처럼 핫 아이템 경합이 몰리면 재시도 폭주** → 오히려 느려짐. 이 약점을 직접 목격하는 것이 학습 포인트(D3 △).

### 옵션 B — 원자적 조건부 UPDATE (`WHERE qty >= n`) ✅ 운영 기본
```sql
UPDATE stocks SET qty = qty - :n WHERE product_id = :id AND qty >= :n;  -- affected rows=0 → 부족
```
- 철학: **검사와 차감이 한 원자 연산 → "충돌"이라는 개념 자체가 없음.**
- 장점: race 자체가 성립 안 함(D1 ◎). 데드락 위험 낮고 락 점유 짧음(D3 ◎). 단일 MySQL에 가장 자연스럽고 단순.
- 단점: 멀티라인 all-or-nothing은 한 트랜잭션 + 하나라도 0행이면 롤백. "왜 실패했나"를 affected rows로만 앎(D2 ○).

### 옵션 C — Redis (DECR/Lua) — 2단계
재고 카운터를 Redis에 두고 원자적 차감. 진실(DB)은 비동기 정합.
- 철학: **진실을 외부 카운터로 빼서 앞단에서 차감** (고동시성 게이팅).
- 장점: 매우 높은 처리량, 핫 아이템·선착순에 강함(D4 ◎).
- 단점: **Redis ↔ DB 이중 소스 정합성** 문제, 장애 복구·영속성 복잡. 단일 인스턴스·학습 규모엔 오버엔지니어링이며, PV-1(가용재고=물리재고)와 이중화가 충돌(D3 ✗).

## 4. 결정 (Decision Outcome)

**재고 차감을 아웃바운드 포트로 추상화하고, 네 기법을 어댑터로 구현해 비교한다.** 운영 기본은 **B**.

### 4.1 포트 추상화 (헥사고날 활용)
```
application:   interface StockDeducer { DeductResult deduct(OrderId, List<Line>); }   ← 아웃바운드 포트
                    ▲              ▲                 ▲                ▲
infrastructure: Pessimistic…   Optimistic…   AtomicUpdate…(default)  Redis…           ← 어댑터 4개
```
- 도메인·정책(PV-4 불변식)은 **무변경** — 기법은 인프라 어댑터일 뿐이다.
- `@Profile`/설정으로 어댑터를 갈아끼우며 **동일 시나리오를 벤치마크**한다.
- "포트 하나에 어댑터 여럿"으로 헥사고날의 진가를 직접 체득(ADR-0005 패키지 구조 위에서).

### 4.2 단계 구분
| 단계 | 어댑터 | 비용 |
|------|--------|------|
| **1 (순수 DB)** | A 비관적 락 · D 낙관적 락 · B 원자적 UPDATE | 추가 인프라 0 → 먼저 |
| **2 (인프라)** | C Redis | Redis 인스턴스 + 이중 소스 정합성(별도 학습 주제) → 분리 |

### 4.3 운영 기본 = B
- PV-4를 **race가 성립하지 않는 구조**로 보장(검사·차감이 한 연산) — 가장 단순·확실(D1).
- 데드락·재시도 폭주·이중 소스 같은 부수 위험이 가장 적다(D3). 단일 MySQL에 자연스럽다.
- 실무에서도 평상시 재고 차감의 디폴트가 B다. (A는 복잡한 임계영역, C는 선착순/플래시세일용)

### 4.4 공통 동시성 테스트 하니스
`N개 스레드가 재고 1개에 동시 차감 → 오버셀 재현(E5) → 같은 테스트를 4개 어댑터에 돌려 "오버셀 0" 증명`.
- 가장 가치 있는 학습 산출물: "race가 B에선 미성립 / A·D에선 막힘(블로킹 vs 재시도) / C에선 외부 차감"을 **눈으로 비교**.
- 재시도(PT-2) 결합 시 차감은 멱등해야 한다 — 동일 이벤트 중복 소비는 Inbox(PI-5)로 1회만 처리되어 중복 차감이 막힌다(PV-5·PT-3 연계).

## 5. 결과 / 영향 (Consequences)

**긍정**
- PV-4 오버셀 불변식을 4가지 방식으로 보장·비교 → 동시성 제어의 *철학적 스펙트럼*(블로킹/논블로킹/원자/외부)을 한 번에 체득.
- 포트 추상화로 도메인 무변경 + 어댑터 교체 가능 → 헥사고날 설계가 실제로 작동함을 확인.
- 공통 테스트 하니스가 회귀·비교의 단일 기준이 된다.

**부정 / 주의**
- 4개 구현은 단일 기법보다 **선행 작업량이 크다** — 단계 분리로 완급 조절(1단계 DB 3종 → 2단계 Redis).
- (B) 멀티라인 차감은 단일 트랜잭션 경계를 지켜 부분 성공을 막아야 함(PV-3).
- (A) `product_id` 정렬 잠금으로 데드락 회피. (D) 재시도 상한·백오프 필요(무한 재시도 금지).
- (C) Redis↔DB 정합성은 그 자체가 학습 주제 → 2단계에서 write-through/최종일관성 설계 별도.
- `inventory.infrastructure`의 `spring-boot-starter-data-redis` 의존은 **2단계까지 미사용** — 그대로 보류한다(제거하지 않음, C 구현 전제).

## 6. 미해결 (이 ADR 범위 밖)

- `stocks` 테이블 스키마: 낙관적 락용 `version` 컬럼은 D 어댑터에서만 쓰는데 공용 테이블에 둘지(타 어댑터는 무시) vs 분리 — 첫 구현 시 결정.
- C(Redis) 2단계의 정합성 모델(write-through vs write-behind, 장애 시 진실 복원) → 필요 시 별도 ADR.
- 벤치마크 측정 지표(처리량·p99·오버셀 0 검증·재시도 횟수) 표준 — 테스트 하니스 작성 시 확정.
