# ADR-0005: 헥사고날 레이어 — 별도 모듈 vs 패키지

- **상태(Status)**: Accepted
- **날짜**: 2026-06-27
- **관련 문서**: [architecture.md](../architecture.md) §1·§2·§6·§7, [ADR-0001](./0001-saga-orchestration-vs-choreography.md)
- **선행 결정**: 초기 골격(9b7eca6)은 컨텍스트당 3개 레이어 모듈(총 11모듈)로 구성 → 본 ADR로 변경

---

## 1. 맥락 (Context)

초기 빌드 골격은 바운디드 컨텍스트(order·payment·inventory)마다 헥사고날 3계층을 **각각 별도 Gradle 모듈**(`*-domain`, `*-application`, `*-infrastructure`)로 분리했다. shared·bootstrap 포함 **총 11개 모듈**.

이 구조는 레이어 의존 규칙(A-1·A-3·A-6)을 *컴파일타임*에 강제하는 장점이 있으나, 아직 비즈니스 코드가 0인 시점에 모듈 11개·빌드 파일 12개의 골격을 떠안는다.

> 이 프로젝트의 목적은 **EDA·DDD 학습**(todo.md)이다. "무엇을 모듈로 가를 것인가"가 학습 초점을 좌우하므로 코드 작성 전에 정리한다.

핵심 질문: **헥사고날 레이어 분리는 DDD 학습에 꼭 모듈이어야 하는가, 패키지로 충분한가?**

---

## 2. 결정 기준 (Decision Drivers)

| # | 기준 | 가중치 |
|---|------|--------|
| D1 | **학습 초점 적합성** — EDA·DDD의 본질(컨텍스트 경계·이벤트·Aggregate)에 집중하는가, 직교한 ceremony를 줄이는가 | 높음 |
| D2 | **경계 강제력 유지** — 레이어 규칙·컨텍스트 격리가 자동으로 강제되는가 | 높음 |
| D3 | **응집도 / 변경 비용** — 한 컨텍스트의 변경이 한 곳에서 완결되는가 | 중간 |
| D4 | **구조 복잡도** — 모듈/빌드 파일 수, 초기 인지 부담 | 중간 |
| D5 | **마이크로서비스 전환 적합성** — 나중에 컨텍스트를 독립 배포로 쪼개기 쉬운가 | 중간 |

## 3. 고려한 옵션 (Considered Options)

### 옵션 A — 레이어를 패키지로 (컨텍스트당 단일 모듈) ✅ 채택

컨텍스트 = 모듈. 그 안에서 domain·application·infrastructure를 **패키지**로 나눈다. 총 5모듈(shared·order·payment·inventory·bootstrap).

**장점**
- **레이어 분리는 헥사고날/클린 아키텍처지 DDD가 아니다** — 모듈로 강제하지 않아도 DDD 학습(컨텍스트·이벤트·Aggregate)에 지장 없음 (D1 ◎)
- 컨텍스트 변경이 한 모듈 안에서 완결 → 응집도↑ (D3 ◎)
- 모듈 5개·빌드 파일 6개로 인지 부담↓ (D4 ◎)
- **레이어 규칙(A-1·A-3·A-6)은 ArchUnit이 패키지 기준으로 그대로 강제** — 기존 테스트가 이미 `..domain..` 패키지 기준이라 무변경으로 동작 (D2 ○)

**단점**
- 세 계층이 같은 모듈/클래스패스에 있어 **도메인 순수성을 컴파일러가 막지 못한다** → 전적으로 ArchUnit(테스트)에 의존 (D2 △ — 컴파일 강제 → 테스트 강제로 강등)

### 옵션 B — 레이어를 모듈로 (컨텍스트당 3모듈) — 기존 골격

**장점**
- domain 모듈에 프레임워크 의존이 *물리적으로 없으므로* `@Entity` import 자체가 **컴파일 에러** → 도메인 순수성을 컴파일타임에 보장 (D2 ◎)

**단점**
- 레이어-모듈 분리는 DDD가 아니라 헥사고날 ceremony → 학습 초점과 직교 (D1 ✗)
- 모듈 11개·빌드 파일 12개. 비즈니스 코드 0 시점에 구조 과투자 (D4 ✗)
- 한 컨텍스트 변경이 3모듈에 흩어짐 (D3 △)

> D5(마이크로서비스 전환)는 두 옵션 모두 동일하게 충족한다 — **쪼개기의 단위는 "레이어"가 아니라 "컨텍스트"**이고, 컨텍스트 경계(C-1)는 두 옵션 다 모듈로 강제하기 때문이다.

---

## 4. 결정 (Decision Outcome)

**옵션 A — 레이어를 패키지로, 컨텍스트당 단일 모듈을 채택한다.** (총 5모듈)

근거:
- **학습 초점**: 목표가 EDA·DDD다. DDD의 본체는 *바운디드 컨텍스트·통합 이벤트·Aggregate*이지 레이어-모듈 분리가 아니다. 직교한 ceremony를 걷어내 분산 패턴(Saga·멱등·동시성) 학습에 자원을 집중한다(D1).
- **경계는 그대로 강제된다**: 잃는 것은 "도메인 순수성의 컴파일타임 보장" 하나뿐인데, 이는 ArchUnit(A-1, 패키지 기준)이 빌드를 깨뜨려 막는다. "컴파일 에러 vs 테스트 실패"의 차이일 뿐 둘 다 빌드를 깨므로 실질 안전성은 동등(D2).
- **컨텍스트 격리는 유지**: 컨텍스트 간 의존 금지(C-1·A-4)는 *모듈 경계*로 계속 컴파일타임 강제된다 — 이건 DDD의 본질이라 절대 양보하지 않는다.
- **전환성 동일**: 나중에 컨텍스트를 독립 마이크로서비스로 쪼갤 때, 분리 단위는 모듈(=컨텍스트)이므로 옵션 A도 손해가 없다(D5).

---

## 5. 결과 / 영향 (Consequences)

**긍정**
- 모듈 11 → 5, 빌드 파일 12 → 6. 구조 인지 부담 감소.
- 컨텍스트별 변경이 단일 모듈 안에서 완결 → 응집도 상승.
- 레이어 규칙·컨텍스트 격리 모두 강제력 유지(ArchUnit + 모듈 경계).

**부정 / 주의 (= 의식적으로 지켜야 할 포인트)**
- **도메인 순수성이 ArchUnit에 100% 의존**한다 → `ModuleBoundaryTest`(A-1)를 CI/빌드에서 **반드시** 실행해야 한다. 테스트를 끄면 도메인이 조용히 오염될 수 있다.
- 같은 모듈 안에 Spring·JPA·Kafka가 클래스패스로 올라오므로, 도메인 패키지에서 *실수로* 프레임워크를 import해도 IDE 자동완성에는 뜬다 → 규율은 ArchUnit이 사후에 잡는다(컴파일이 즉시 막지 않음).

**후속 영향**
- architecture.md §1·§2·§6·§7을 5모듈/패키지 기준으로 갱신(완료, 361a6b2).
- 향후 도메인 코드 추가 시 ArchUnit A-1~A-6이 실제 위반을 잡는지 첫 Aggregate 작성 시점에 재확인.
