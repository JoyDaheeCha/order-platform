# ADR-0001: Saga 구현 방식 — 오케스트레이션 vs 코레오그래피

- **상태(Status)**: Accepted
- **날짜**: 2026-06-26 (결정 변경: 2026-06-27 — 코레오그래피로 변경)
- **관련 정책**: [policy.md](../policy.md) §1, §8(PB), §9(PT)

---

## 1. 맥락 (Context)

주문 도메인은 `Order → Payment → Inventory` 세 컨텍스트에 걸친 분산 트랜잭션이다. 단일 DB 트랜잭션으로 묶을 수 없으므로 **Saga**로 정합성을 맞춘다(정책 §8 보상 트랜잭션). 모듈 간 통신은 **Kafka 메시지 only**(직접 메서드 호출 금지).

Saga 구현에는 두 갈래가 있다. **이 선택이 모듈 구조·코드 골격·보상/타임아웃 로직의 위치를 결정**하므로 코드 작성 전에 못 박는다.

> 이 프로젝트의 목적은 제품 출시가 아니라 **분산 패턴의 깊은 이해**(todo.md)다. 따라서 결정 기준에서 "학습 가치"의 가중치가 가장 높다.

---

## 2. 결정 기준 (Decision Drivers)

| # | 기준 | 가중치 |
|---|------|--------|
| D1 | **학습 가치** — 보상·멱등성·타임아웃을 *명시적으로* 학습 가능한가 (todo.md 핵심) | 높음 |
| D2 | **흐름 가시성 / 디버깅** — Saga 진행 상태와 실패 지점을 추적하기 쉬운가 | 높음 |
| D3 | **결합도** — 컨텍스트 간 느슨한 결합(EDA 원칙) | 중간 |
| D4 | **구현 복잡도** — 학습 범위에서 감당 가능한가 | 중간 |
| D5 | **확장성** — 단계 추가/변경이 쉬운가 | 낮음(학습 범위 고정) |

## 3. 고려한 옵션 (Considered Options)

### 옵션 A — 코레오그래피 (Choreography) ✅ 채택

각 컨텍스트가 이벤트를 구독해 **자율적으로** 다음 동작을 수행한다. 중앙 조정자 없음.

```
OrderPlaced ─▶ [Payment] ─ PaymentCompleted ─▶ [Inventory] ─ StockDeducted ─▶ [Order] confirm
                  │                                  │
            PaymentFailed ─▶ [Order] cancel    StockShortage ─▶ [Payment] refund ─▶ [Order] cancel
```

**장점**
- 순수 EDA, 컨텍스트 간 결합 최소 (D3 ◎)
- 인프라 단순 — 이벤트 토픽만 있으면 됨 (D4 부분 ◎)

**단점**
- **Saga 흐름이 코드 어디에도 한눈에 안 보임** — 여러 컨텍스트의 이벤트 핸들러에 흩어짐 (D2 ✗)
- **보상 로직이 분산** — "지금 어디까지 진행됐고 무엇을 되돌려야 하나"를 각 서비스가 부분만 안다 (D1 ✗ — 보상을 *명시적으로* 배우기 어려움)
- 타임아웃(PT-1) 주체가 모호 — 누가 "결제가 안 온다"를 감지하나
- 단계 늘면 이벤트 의존 그래프가 사이클·스파게티가 되기 쉬움

### 옵션 B — 오케스트레이션 (Orchestration) — 후속 학습 과제

**Saga Orchestrator**(주문 컨텍스트 내 별도 컴포넌트)가 상태 기계로서 각 단계를 **커맨드 메시지**로 지시하고 결과 이벤트를 기다린다. 통신은 여전히 Kafka 비동기 메시지(직접 호출 아님).

```
        ┌─────────── OrderSagaOrchestrator (상태 기계) ───────────┐
        │  ProcessPayment ▶  ◀ PaymentCompleted/PaymentFailed     │
        │  DeductStock    ▶  ◀ StockDeducted/StockShortage        │
        │  (실패 시) RefundPayment ▶ , confirm/cancel Order        │
        └──────────────────────────────────────────────────────────┘
```

**장점**
- **Saga 전체 흐름 + 보상 + 타임아웃이 한 곳(상태 기계)에 명시적으로 모임** (D1 ◎, D2 ◎) — *바로 이 프로젝트가 배우려는 것*
- 실패 시 "어디까지 했고 무엇을 보상할지"가 오케스트레이터 상태로 명확 (PB-1 역순 보상 구현이 자연스러움)
- 타임아웃(PT-1)의 주체가 명확 — 오케스트레이터가 단계별 데드라인 관리
- 디버깅·테스트가 쉬움 (상태 전이를 단위 테스트로 검증)

**단점**
- 오케스트레이터가 흐름을 알아야 함 → 컨텍스트 간 결합이 코레오그래피보다 높음 (D3 △)
  - 단, **데이터 결합이 아니라 "흐름 지식"의 결합**이고, 통신은 여전히 메시지 기반이라 배포 독립성은 유지
- 커맨드 토픽 + 리플라이(이벤트) 토픽으로 토폴로지가 조금 더 복잡 (D4 △)

---

## 4. 결정 (Decision Outcome)

**옵션 A — 코레오그래피를 채택한다.** 중앙 오케스트레이터 없이, 각 컨텍스트가 도메인 이벤트를 구독해 자율적으로 다음 동작을 수행한다.

> **결정 변경 이력**: 최초(2026-06-26)에는 학습 가시성(D1·D2)을 근거로 오케스트레이션(옵션 B)을 제안했으나, 다음 근거로 코레오그래피(옵션 A)로 변경한다.

근거:
- **규모 적합성**: 단계가 3개(결제·재고·확정)로 적고 흐름이 선형적 → 코레오그래피로 운영 가능한 규모(D4 ◎). 중앙 조정자는 이 규모에선 오버엔지니어링.
- **학습 순서 ("문제를 먼저 겪고 해법을 배운다")**: 코레오그래피로 짜면서 *흐름 비가시성·보상 분산·타임아웃 감지 주체 모호*라는 고통을 **직접 체감**하는 것이, 이후 오케스트레이션을 배울 때 그 존재 이유를 몸으로 이해하게 한다. → 오케스트레이션은 **후속 학습 과제**(동일 도메인 재구현).
- **도메인 격리**: 컬리 실무 경험과 동일하게 "도메인이 서로를 모르게" 두는 순수 EDA를 먼저 체화(D3 ◎).
- D1(학습 가치)의 *보상·멱등·타임아웃*은 코레오그래피에서도 다룬다 — 위치가 분산될 뿐 개념은 동일하며, 분산되기 때문에 오히려 더 깊이 신경 써야 한다(아래 §5 주의).

## 5. 결과 / 영향 (Consequences)

**긍정**
- 컨텍스트 간 결합 최소 — 새 구독자(추천·알림 등) 추가 시 발행자 무변경.
- 인프라 단순 — 이벤트 토픽만으로 구성(커맨드/리플라이 토픽 불필요).
- 순수 EDA·도메인 격리를 정석대로 학습.

**부정 / 주의 (= 의식적으로 설계해야 할 학습 포인트)**
- **흐름 비가시성**: Saga 진행 상태가 토픽에 흩어진다 → `orderId`를 전 이벤트의 상관관계 키로 강제(정책 PI-4)하고, 필요 시 주문 상태 조회용 read model을 둔다.
- **보상 분산**: 각 컨텍스트가 실패 이벤트를 구독해 자기 보상을 수행한다 → "어떤 실패 이벤트에 누가 반응하는가"를 명시적으로 문서화(§1 보상 흐름).
- **타임아웃 감지 주체 부재(PT-1)**: 중앙 감시자가 없다 → **Order 컨텍스트에 데드라인 체커**(스케줄러/지연 메시지)를 두어, 기한 내 종료 이벤트가 없으면 취소를 트리거한다.
- **사용자 취소 경합(PC-4)**: 중앙 Saga 상태가 없어 "어디까지 진행됐나"를 Order가 자체 추적해야 한다 → Order Aggregate가 수신 이벤트로 진행도를 누적(*경량 상태 추적*). 이 지점이 "코레오그래피로 횡단 관심사를 다루다 보면 오케스트레이터를 재발명하게 된다"는 핵심 학습 포인트.

**후속 ADR 영향**
- **ADR-0003**(타임아웃/DLQ): 중앙 오케스트레이터가 없으므로 **Order 측 데드라인 체커**의 구현 메커니즘(지연 메시지 vs 스케줄러) 결정 필요.
- **ADR-0004**(스키마/Outbox/Inbox): Saga 상태 테이블 대신, 각 컨텍스트의 Inbox + Order의 진행도 추적/상태 read model 설계.

---

## 6. 미해결 (이 ADR 범위 밖)

- Order 데드라인 체커의 구현 메커니즘(지연 메시지 vs 스케줄러) → ADR-0003
- 주문 상태 가시성을 위한 read model 도입 여부·범위 → ADR-0004
- (후속 학습) 동일 도메인을 오케스트레이션으로 재구현해 트레이드오프 비교
