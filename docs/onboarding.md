# onboarding — order-platform 처음 만나기

> **이 문서는 누구를 위한 것인가**
> 이 프로젝트에 처음 합류했거나, "분산 시스템·Kafka·Saga 같은 말은 들어봤는데 정확히 뭔지 모르겠다"는 사람을 위한 글이다.
> 코드를 읽기 **전에** 이 문서를 먼저 읽으면, 왜 이렇게 만들었는지가 보인다. 전문 용어는 비유로 먼저 설명하고, 정확한 정의는 뒤에 붙인다.

---

## 1. 한 문장으로

> **온라인 쇼핑몰에서 "주문하기"를 눌렀을 때, 결제와 재고가 서로 다른 시스템에 흩어져 있어도 돈과 재고가 어긋나지 않게 맞추는 방법**을 공부하는 프로젝트.

실제 상품을 파는 게 목적이 아니라, **그 뒤에서 벌어지는 일을 제대로 이해하는 것**이 목적이다. 그래서 회원가입·장바구니·배송 같은 건 일부러 다 뺐다([product-spec.md](./product-spec.md)).

---

## 2. 왜 이게 어려운 문제인가 (핵심 직관)

은행 계좌 이체를 생각해보자. "내 통장에서 빠지고" + "상대 통장에 들어가고"가 **둘 다 되거나 둘 다 안 되거나** 해야 한다. 하나만 성공하면 돈이 사라지거나 복사된다.

쇼핑몰 주문도 똑같다. 주문 하나가 성립하려면 세 가지가 다 맞아야 한다.

1. **주문**이 만들어지고
2. **결제**가 되고
3. **재고**가 차감된다

문제는 이 셋이 **서로 다른 시스템(서비스)** 에 있다는 것이다. 옛날 방식대로 "하나의 데이터베이스 트랜잭션으로 묶어서 한 번에 처리"가 **불가능**하다. 결제 시스템과 재고 시스템은 남남이라 같은 트랜잭션에 못 들어간다.

그럼 어떻게 "셋 다 되거나 셋 다 취소"를 보장할까? → 이 질문의 답이 이 프로젝트의 전부다.

## 3. 핵심 개념을 비유로

### 🤝 Saga (사가) — "릴레이 경주"

하나의 큰 작업을 한 번에 못 하니, **작은 단계로 쪼개서 바통을 넘기듯** 진행한다.

> 주문 생성 → (바통) → 결제 → (바통) → 재고 차감 → 완료!

각 단계는 자기 일만 하고 "끝났다"고 외친다. 그러면 다음 주자가 이어받는다. 이걸 **Saga(사가)** 라고 부른다.

이 프로젝트는 그중에서도 **코레오그래피(choreography, 안무) 방식**을 쓴다. 지휘자가 없는 **군무**를 떠올리면 된다. 각 댄서(서비스)가 "옆 사람이 이 동작을 하면 나는 이 동작을 한다"는 약속만 알고 자율적으로 춤춘다. 중앙에서 "너 이제 결제해, 너 이제 재고 빼"라고 지시하는 감독이 없다. ([ADR-0001](./adr/0001-saga-orchestration-vs-choreography.md)에서 왜 이 방식을 골랐는지 자세히 적었다.)

### ↩️ 보상 트랜잭션 (Compensation) — "되돌리기"

릴레이 도중에 한 주자가 넘어지면? 이미 지나온 단계를 **거꾸로 되감아야** 한다.

> 예: 결제는 됐는데 재고가 부족하다 😱
> → 이미 받은 돈을 **환불**해서 원래대로 돌린다.

이렇게 "이미 한 일을 거꾸로 되돌리는 보정 작업"을 **보상 트랜잭션**이라 한다. 일반 트랜잭션처럼 "없던 일로(rollback)" 만들 수 없으니, **반대 동작을 명시적으로 실행**해서 결과적으로 맞춘다. (정책 §8 PB)

### 📮 이벤트 (Event) — "방송으로 사실을 알리기"

서비스끼리 직접 전화("야 결제해줘")하지 않는다. 대신 **사내 방송으로 사실을 알린다**.

> "📢 주문이 생성되었습니다 (`OrderPlaced`)"
> → 결제 서비스가 그 방송을 듣고 알아서 결제를 시작한다.

이 "방송"이 **이벤트**이고, 방송 시스템이 **Kafka(카프카)** 다. 핵심은: **방송하는 쪽은 누가 듣는지 모른다.** 그래서 나중에 "알림 서비스"를 추가해도 방송하는 쪽 코드를 안 고쳐도 된다. 이게 느슨한 결합(EDA, 이벤트 드리븐 아키텍처)의 힘이다.

### 🔑 멱등성 (Idempotency) — "여러 번 눌러도 한 번만"

엘리베이터 버튼을 다섯 번 눌러도 엘리베이터는 한 대만 온다. 이게 **멱등성**이다.

네트워크는 불안정해서 같은 요청이 **두 번 도착**할 수 있다. "결제하기"를 눌렀는데 응답이 늦어 한 번 더 누르면? 결제가 두 번 되면 안 된다.

> 그래서 모든 주문 요청에 **멱등키(고유 번호표)** 를 붙인다. 같은 번호표가 또 오면 "어, 이미 처리했네" 하고 **기존 결과만 돌려준다**. (정책 §3 PI)

이건 두 군데서 필요하다:
- **요청을 보내는 쪽** (사용자의 중복 클릭) → 멱등키
- **방송을 듣는 쪽** (Kafka가 같은 방송을 두 번 전달) → **Inbox 패턴**("이 방송 들은 적 있나?" 장부에 체크)

### 📦 재고 동시성 — "마지막 한 개를 두 명이 동시에"

재고가 딱 1개 남았는데 두 명이 **동시에** 주문하면? 잘못 만들면 둘 다 성공해서 재고가 -1이 된다(오버셀). 이걸 막는 게 **동시성 제어**다. (정책 §6 PV-4, [ADR-0002](./adr/0002-inventory-concurrency.md))

## 4. 주문 하나가 처리되는 과정 (해피 패스 스토리)

손님이 디자이너 가방을 1개 주문한다고 하자.

```
1. 손님: "이거 주문할게요" (멱등키 #abc 동봉)
        │
        ▼
2. 주문 서비스: 주문서를 PENDING(결제 대기) 상태로 만든다
   → 📢 "OrderPlaced(주문 생성됨)" 방송
   → 손님에게 즉시 "접수됐어요! (202)" 응답  ← 여기서 손님과의 통화는 끝!
        │                                       (나머지는 뒤에서 알아서 진행)
        ▼
3. 결제 서비스: 방송 듣고 결제 시도 → 성공
   → 📢 "PaymentCompleted(결제 완료)" 방송, 주문은 PAID 상태로
        │
        ▼
4. 재고 서비스: 방송 듣고 재고 1개 차감
   → 📢 "StockDeducted(재고 차감됨)" 방송
        │
        ▼
5. 주문 서비스: 방송 듣고 주문을 CONFIRMED(확정)로 전환 🎉
        │
        ▼
6. 손님: "내 주문 어떻게 됐지?" 하고 가끔 새로고침(폴링)하면
   → "CONFIRMED!" 확인
```

**왜 2번에서 바로 응답하고 끝낼까?** 결제·재고까지 다 기다리면 손님은 화면 앞에서 몇 초씩 멈춰 있어야 한다. 대신 "접수했어요"라고 먼저 답하고, 나머지는 뒤에서 비동기로 처리한다. 손님은 나중에 상태를 확인하면 된다. (이 결정의 근거는 [ADR-0006](./adr/0006-inbound-api-response-and-idempotency.md))

주문서의 상태는 이렇게만 흐른다 (다른 경로는 전부 금지 — 정책 §4 PS):

```
[PENDING] ──결제완료──▶ [PAID] ──재고차감──▶ [CONFIRMED] (성공 종료)
    │                     │
    │                     └─ 재고부족 ─▶ [CANCELLED] (환불 후 취소)
    └─ 결제실패 / 사용자취소 ──────────▶ [CANCELLED]
```

## 5. 잘못되면 어떻게 되나 (보상 스토리)

가장 흥미로운 케이스: **결제는 됐는데 재고가 부족**할 때.

```
결제 완료(돈 빠짐) ✅  →  재고 차감 시도... 어, 품절! ❌
        │
        ▼
재고 서비스: 📢 "StockShortage(재고 부족)" 방송
        │
        ▼
결제 서비스: 방송 듣고 → 받은 돈 전액 환불 (RefundPayment)
   → 📢 "PaymentRefunded(환불 완료)" 방송
        │
        ▼
주문 서비스: 주문을 CANCELLED로 전환
```

핵심 포인트:
- **이미 한 일(결제)만 거꾸로 되돌린다.** 재고는 차감 자체가 안 됐으니 되돌릴 것도 없다. (정책 PB-1: "수행한 단계의 역순으로만 보상")
- **보상은 절대 유실되면 안 된다.** 환불이 한 번 실패해도 재시도하고, 그래도 안 되면 따로 모아두고(DLQ) 끝까지 처리한다. 돈/재고가 걸린 문제라 "그냥 포기"가 없다. (정책 PB-3)

그 외 실패 시나리오들은 [product-spec.md §5](./product-spec.md)에 표로 정리돼 있다 (결제 실패, 중복 주문, 사용자 취소 등).

## 6. 코드는 어떻게 생겼나 (구조 쉽게 보기)

### 큰 그림: 한 덩어리지만 칸막이가 있다

이 프로젝트는 **모듈러 모노리스(Modular Monolith)** 다. 쉽게 말하면:

> **하나의 큰 집이지만, 방마다 벽이 튼튼하게 서 있다.**

진짜 마이크로서비스(집 여러 채)는 운영이 복잡하다. 그래서 **집은 한 채(단일 배포)** 로 두되, 방(서비스) 사이의 벽을 마이크로서비스처럼 단단하게 막아서 **나중에 분리하기 쉽게** 해둔다.

```
order-platform/  (집 한 채)
├── order/        🛒 주문 방 — 모든 게 여기서 시작
├── payment/      💳 결제 방
├── inventory/    📦 재고 방
├── shared/       📢 공용 게시판 (방송 메시지 양식만 둔다)
└── bootstrap/    🔌 전원 스위치 (실제로 프로그램을 켜는 곳)
```

**철칙**: order 방은 payment 방의 물건을 직접 가져다 쓸 수 없다. 오직 **공용 게시판(shared)을 통한 방송(Kafka 이벤트)** 으로만 소통한다. (아키텍처 규칙 C-1, C-2)

> 이 규칙을 사람이 실수로 어길 수 있으니, **ArchUnit이라는 테스트가 자동으로 감시**한다. 벽을 뚫는 코드를 쓰면 테스트가 실패한다. ([architecture.md §6](./architecture.md))

### 방 하나의 내부: 3개의 층

각 방은 다시 세 층으로 나뉜다. **순수한 규칙일수록 안쪽**에 둔다.

| 층 | 비유 | 하는 일 | 기술에 의존? |
|----|------|---------|------|
| `domain` | 🧠 **두뇌 (순수 규칙)** | "주문은 1개 이상 라인이 있어야 한다" 같은 변하지 않는 비즈니스 규칙 | ❌ 아무것에도 (순수 Java) |
| `application` | 🎬 **감독 (시나리오)** | "주문을 만들려면 이 순서로 처리한다"는 흐름 조율 | ⚠️ 최소한만 |
| `infrastructure` | 🔌 **콘센트 (바깥 연결)** | DB 저장, Kafka 방송, 웹 요청 받기 등 실제 바깥 세상과의 연결 | ✅ Spring, JPA, Kafka |

**왜 이렇게 나눌까?** 두뇌(domain)가 특정 기술(Kafka, MySQL)을 모르게 하면, 나중에 "MySQL을 다른 DB로 바꾸자"가 되어도 두뇌는 그대로 둘 수 있다. 비즈니스 규칙이 기술 변화로부터 보호된다. 이게 **헥사고날 아키텍처**의 핵심이다. ([architecture.md §2](./architecture.md))

의존 방향은 항상 **바깥 → 안쪽** 한 방향:
```
infrastructure(콘센트) ──▶ application(감독) ──▶ domain(두뇌)
```
두뇌는 콘센트를 모른다. 콘센트가 두뇌를 안다.

## 7. 핵심 용어 사전 (빠른 참조)

| 용어 | 한 줄 뜻 | 비유 |
|------|----------|------|
| **Saga** | 분산된 단계를 이어 처리하는 방식 | 릴레이 경주 |
| **코레오그래피** | 지휘자 없이 각자 이벤트 듣고 반응 | 군무 |
| **보상 트랜잭션** | 이미 한 일을 반대 동작으로 되돌리기 | 환불 |
| **이벤트 / Kafka** | "사실"을 방송하고 듣는 구조 | 사내 방송 |
| **멱등성** | 여러 번 해도 결과는 한 번과 같음 | 엘리베이터 버튼 |
| **멱등키** | 중복을 가려내는 요청 고유 번호표 | 대기표 |
| **Inbox 패턴** | 받은 방송을 두 번 처리 안 하게 체크 | 출석부 |
| **Outbox 패턴** | DB 저장과 방송을 안전하게 묶기 | 발송 대기함 |
| **동시성 제어** | 동시 주문에도 재고가 안 깨지게 | 마지막 1개 쟁탈전 정리 |
| **Aggregate** | 함께 변해야 하는 데이터+규칙 묶음 | 주문서 한 장 |
| **모듈러 모노리스** | 한 덩어리지만 내부 경계가 단단함 | 칸막이 있는 집 |
| **헥사고날** | 핵심 규칙을 기술로부터 격리 | 두뇌-감독-콘센트 |
