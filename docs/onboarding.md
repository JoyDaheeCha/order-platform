# onboarding — order-platform 지금 어디까지 와 있나

> **이 문서는 누구를 위한 것인가**
> 이 프로젝트에 처음 합류해 "설계는 끝난 건가? 지금 뭘 하면 되나?"가 궁금한 사람을 위한 글이다.
> 설계 *결정*은 ADR로 거의 못박혔지만, 그걸 **코드로 옮기는 일은 이제 막 골격만 세운 상태**다. 이 문서가 그 현재 좌표를 알려준다.
>
> ⚠️ 이 문서는 **시점 스냅샷**이다. 구현이 진행되면 빠르게 낡으므로, 큰 변화가 있을 때 갱신한다.

---

## 0. 한 줄 결론

> **"무엇을·왜 만들지에 대한 의사결정(논리 설계)은 끝났고, 그걸 코드 구조로 옮기는 일(구현)은 이제 막 골격만 세운 상태"** 다.

"문서가 잘 정리돼 있으니 설계가 끝난 것 같다"는 인상을 받기 쉽지만, **설계 전체로 보면 아직 안 끝났다.** 아래에서 정확한 상태를 보여준다.

---

## 1. 지금 어디까지 와 있나

### ✅ 끝난 것 — 설계 문서 (Design Docs)

- product-spec / policy / architecture / design / (이) onboarding 문서 + **ADR 7건 전부 Accepted**
- product-spec §7의 미해결 질문들도 전부 `[x]` 해소 완료
  (Saga 방식, PAID/CONFIRMED 분리, 가짜 PG 실패 주입, 재고 동시성)

### 🟡 시작만 된 것 — 코드 골격 (Skeleton)

Java 파일이 13개 있지만, 내용을 보면 거의 다 **빈 껍데기**다:

```
- package-info.java × 9        ← 패키지 선언만 (빈 파일)
- IntegrationEvent.java        ← shared 마커 인터페이스 수준
- ModuleBoundaryTest.java      ← ArchUnit 경계 테스트 (골격 강제용)
- OrderPlatformApplication.java ← @SpringBootApplication main
```

즉 **5개 Gradle 모듈 + 패키지 구조 + 빌드 스크립트 + ArchUnit 테스트**까지는 깔렸지만,
**실제 도메인/비즈니스 로직은 아직 한 줄도 없다.**

### ❌ 아직 안 된 것 — 구현 전체

- Order/Payment/Inventory의 Aggregate · VO · 상태 전이 (domain)
- 유스케이스 · 포트 (application)
- JPA · Kafka 컨슈머/프로듀서 · REST · Outbox/Inbox 어댑터 (infrastructure)
- read model, 데드라인 체커, 재고 동시성 4어댑터
- 통합 이벤트 payload 레코드들 (shared)
- 테스트 (동시성 · 멱등성 · 보상 시나리오)

## 2. "설계가 끝났나?"에 대한 정확한 답

이 프로젝트의 의도를 보면 이 순서가 자연스럽다. [`design.md`](./design.md)에 적힌 대로 **"문제를 먼저 겪고 해법을 나중에 배운다"**가 핵심 철학이라, 설계 결정과 그 트레이드오프를 ADR로 충분히 못박은 **뒤에** 구현으로 들어가는 구조다. 학습용 프로젝트라서 "왜 이렇게 정했나"를 문서로 남기는 것 자체가 산출물의 절반이다.

그리고 설계도 "영구 동결"은 아니다. ADR은 [`design.md`](./design.md)가 강조하듯 **"의식적으로 미룬 것"**을 일부러 남겨둔다. 예를 들어:

- 오케스트레이션 방식 재구현 (코레오그래피와 비교 학습) — [ADR-0001](./adr/0001-saga-orchestration-vs-choreography.md)
- 단계별 타임아웃 / 워크플로 엔진 — [ADR-0003](./adr/0003-order-deadline-checker.md) §7
- 셀러 · 운영자 페르소나, 재고 예약 모델 등 확장 학습 과제 — [product-spec.md §3](./product-spec.md)

이런 건 **구현하면서 새로 겪는 문제에 따라 ADR이 추가되거나 갱신될** 여지가 있다.
(현재 작업 브랜치 이름도 `feature/add-structure` — "구조 추가" 단계라는 뜻이다.)

## 3. 신규 입사자의 다음 스텝

설계 *문서*는 잘 익었고, 이제 **그 설계를 코드로 구현하는 본 작업**이 남았다. 추천하는 첫걸음:

> **가장 단순한 happy path 한 줄(예: `PlaceOrder` → `OrderPlaced` 발행)을 도메인부터 끝까지 얇게 구현**해본다.

세 컨텍스트를 한꺼번에 채우기보다, 주문 한 건이 흐르는 가장 얇은 경로(walking skeleton)를 먼저 관통시키면 Outbox·Inbox·이벤트 계약이 실제로 어떻게 맞물리는지 몸으로 익힐 수 있다.
