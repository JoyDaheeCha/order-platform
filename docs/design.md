# design — 설계 의사결정 개요 (면접용 설명서)

> **이 문서는 누구를 위한 것인가**
> 이 프로젝트의 설계를 평가하려는 **면접관**을 위한 문서다. "무엇을 만들었나"보다 **"왜 이렇게 결정했고, 무엇을 의식적으로 포기했나"**에 초점을 맞춘다.
>
> 각 결정의 상세 근거는 [`adr/`](./adr)에, 도메인 규칙은 [`policy.md`](./policy.md)에, 물리 구조는 [`architecture.md`](./architecture.md)에 있다. 이 문서는 그 위를 가로지르는 **설계 서사(narrative)**다.

---

## 0. 30초 요약

> **세 개의 바운디드 컨텍스트(주문·결제·재고)에 걸친 분산 트랜잭션을, 단일 DB 트랜잭션 없이 Saga·이벤트 기반으로 정합성 있게 맞추는 주문 도메인.** 모듈러 모노리스로 단일 배포하되, 컨텍스트 경계는 마이크로서비스 수준으로 강제한다.

- **아키텍처**: Modular Monolith + Hexagonal(컨텍스트당 단일 모듈, 레이어는 패키지)
- **분산 정합성**: Choreography Saga + 보상 트랜잭션
- **메시징**: Kafka + Outbox/Inbox 패턴 (effectively-once)
- **데이터**: 컨텍스트별 스키마 분리(단일 MySQL) + 컨텍스트별 DataSource
- **모든 설계 결정은 ADR로 문서화** — 7개 ADR, 각각 *문제 → 옵션 비교 → 결정 → 트레이드오프 → 의식적으로 미룬 것* 구조

> ⚠️ 이 프로젝트는 **제품 출시가 아니라 분산 패턴의 깊은 이해**가 목적이다. 그래서 모든 결정 기준에서 **"학습 가치"의 가중치가 가장 높다.** 이 전제를 알아야 아래 결정들이 일관되게 읽힌다.

## 1. 설계를 관통하는 세 가지 원칙

이 프로젝트의 모든 ADR에는 공통된 사고방식이 깔려 있다. 개별 결정보다 이 **메타 원칙**이 설계 역량의 핵심이다.

### 원칙 1 — "문제를 먼저 겪고, 해법을 나중에 배운다"

가장 중요한 결정인 **Saga 방식**에서 드러난다. 흐름 가시성·디버깅만 보면 **오케스트레이션이 유리**하다(상태 기계 한 곳에 흐름·보상·타임아웃이 모인다). 그런데도 **코레오그래피를 택했다.**

> 단계가 3개·선형이라 코레오그래피로 충분한 규모이기도 하지만, 더 본질적인 이유는 **"흐름 비가시성·보상 분산·타임아웃 감지 주체 모호"라는 고통을 직접 겪어봐야 오케스트레이션의 존재 이유를 몸으로 이해한다**는 것이다. 오케스트레이션은 *후속 학습 과제*로 같은 도메인을 재구현하며 비교한다. ([ADR-0001](./adr/0001-saga-orchestration-vs-choreography.md))

면접 포인트: **"쉬운 길 대신 학습 곡선을 의도적으로 설계했다"** — 정답을 아는 것과, 왜 그게 정답인지 체득하는 것을 구분한다.

### 원칙 2 — "포트로 추상화하고, 여러 구현을 비교한다"

단일 정답을 고르는 대신, **아웃바운드 포트 하나에 여러 어댑터**를 두고 갈아끼우며 비교하는 패턴이 반복된다.

| 비교 대상 | 포트/축 | 어댑터/옵션 | ADR |
|-----------|---------|-------------|-----|
| 재고 동시성 | `StockDeducer` | 비관적 락 / 낙관적 락 / 원자적 UPDATE(기본) / Redis | [0002](./adr/0002-inventory-concurrency.md) |
| 헥사고날 레이어 | 모듈화 단위 | 별도 모듈 vs 패키지(채택) | [0005](./adr/0005-hexagonal-layer-as-package-vs-module.md) |
| Saga | 조정 방식 | 코레오그래피(채택) vs 오케스트레이션(후속) | [0001](./adr/0001-saga-orchestration-vs-choreography.md) |

면접 포인트: **헥사고날 아키텍처가 "장식"이 아니라 실제로 작동함을 증명** — 도메인·정책(PV-4 오버셀 금지) 무변경으로 4가지 동시성 기법을 갈아끼우고, 공통 테스트 하니스로 "오버셀 0"을 4번 검증한다.

### 원칙 3 — "경계는 강제 메커니즘을 명시적으로 고른다"

"경계를 지키자"는 구호가 아니라, **무엇이 무엇을 어떻게 막는지**를 결정마다 정한다. 핵심은 강제력의 3단계를 의식적으로 구분한 것이다.

| 강제 대상 | 메커니즘 | 강제력 | 근거 |
|-----------|----------|--------|------|
| 컨텍스트 간 코드 의존 (C-1) | **모듈 경계** | 컴파일타임 | `order`가 `payment` 패키지를 import하면 컴파일 실패 |
| 레이어 의존 방향 (A-1·A-3·A-6) | **ArchUnit 테스트** | 빌드 타임(테스트) | 같은 모듈이라 컴파일러가 못 막음 → 테스트가 막음 ([ADR-0005](./adr/0005-hexagonal-layer-as-package-vs-module.md)) |
| 컨텍스트 간 데이터 결합 | **컨텍스트별 DataSource** | 물리적 불가능 | 타 스키마 테이블이 EntityManager에 아예 안 보임 ([ADR-0004](./adr/0004-schema-separation-outbox-readmodel.md)) |

면접 포인트: ADR-0005에서 레이어를 모듈로 쪼개면 도메인 순수성이 *컴파일타임에* 보장되지만, **"DDD의 본질은 컨텍스트 경계지 레이어-모듈 분리가 아니다"**라고 판단해 패키지로 강등하고, 잃은 강제력은 ArchUnit으로 메웠다. 11모듈 → 5모듈. **"무엇을 컴파일러에게 맡기고 무엇을 테스트에 맡길지"를 비용 대비로 결정**한 사례.

## 2. 핵심 설계 결정 7선 (ADR 다이제스트)

각 ADR을 *문제 → 결정 → 트레이드오프 → 의식적으로 미룬 것* 한 줄씩으로 압축했다. 깊이 들어갈 ADR은 링크로.

### ① Saga: 코레오그래피 — [ADR-0001](./adr/0001-saga-orchestration-vs-choreography.md)
- **문제**: 3개 컨텍스트 분산 트랜잭션을 단일 DB 트랜잭션으로 못 묶는다.
- **결정**: 중앙 조정자 없이 각 컨텍스트가 이벤트 구독해 자율 반응(코레오그래피).
- **트레이드오프**: 결합 최소·인프라 단순 ↔ **흐름 비가시성·보상 분산·타임아웃 주체 부재**.
- **미룸**: 오케스트레이션 재구현(같은 도메인) → 비교 학습.

### ② 데이터 영속화: 스키마 분리 + Outbox/Inbox + Read Model — [ADR-0004](./adr/0004-schema-separation-outbox-readmodel.md)
세 결정이 **맞물려 도출**되는 게 핵심이다:
- 스키마를 분리하면 → 컨텍스트 간 **JOIN/FK/단일 트랜잭션이 불가능** →
- "주문이 어디까지 진행됐나"를 JOIN으로 못 봄 → **read model**(`order_saga_progress`) 필요 →
- "상태 변경 + 이벤트 발행"을 한 트랜잭션으로 못 묶음(dual-write) → **Outbox** 필요.
- **결정**: 컨텍스트별 스키마(단일 MySQL) + 컨텍스트별 DataSource(B-2). 타 스키마 접근이 *물리적으로* 불가능.
- **트레이드오프**: 경계가 구조로 강제됨 ↔ 컨텍스트당 영속화 `@Configuration` 보일러플레이트 3세트.

### ③ 재고 동시성: 포트 + 4어댑터, 기본은 원자적 UPDATE — [ADR-0002](./adr/0002-inventory-concurrency.md)
- **문제**: 동일 상품 동시 주문 시 read-modify-write 경쟁 → 오버셀(PV-4 위반).
- **결정**: `StockDeducer` 포트 + 4어댑터 비교. 운영 기본 = **원자적 조건부 UPDATE** (`WHERE qty >= n`).
- **왜 B가 기본인가**: 비관적 락(블로킹)·낙관적 락(재시도 폭주)·Redis(이중 소스)와 달리, **검사와 차감이 한 원자 연산이라 "충돌"이라는 개념 자체가 없다.** 데드락·재시도 위험 최소.
- **트레이드오프**: 4구현은 선행 작업량↑ → 1단계(순수 DB 3종) / 2단계(Redis) 분리.

### ④ 타임아웃 + 재시도/DLQ: 데드라인 체커 — [ADR-0003](./adr/0003-order-deadline-checker.md)
- **문제**: 코레오그래피엔 "결제가 안 온다"를 지켜볼 **중앙 감시자가 없다**(ADR-0001의 빚).
- **결정 A (입도)**: **사가 전체 타임아웃**(단계별 ❌). 단계별로 가면 Order가 in-flight 단계 상태를 모델링해야 하고, 그건 **오케스트레이터의 본체**라 ADR-0001이 미룬 걸 앞당겨 중복.
- **결정 B (메커니즘)**: **폴링 스위퍼**(지연 메시지 ❌). **Kafka엔 네이티브 지연 메시지가 없어서**, durable하게 만들려는 모든 변형이 결국 "상태 테이블 + 폴러"로 수렴한다.
- **결정 C (재시도)**: **비즈니스 실패와 인프라 오류를 절대 안 섞는다.** 재고부족·결제거절 = 재시도 없이 즉시 보상(정상 분기). DB 순단 등 = 지수 백오프 → DLQ. 재고부족을 예외→재시도로 흘리면 "정상 결과가 DLQ를 오염".

### ⑤ shared 통합 이벤트 계약 — [ADR-0007](./adr/0007-integration-event-contract.md)
- **결정**: `EventEnvelope<T>`(메타 봉투) + payload 분리. `<context>.events` 3토픽. 파티션 키 = `orderId`. 가산적 버저닝(Schema Registry 없음).
- **봉투가 ADR-0004 Outbox 컬럼과 1:1 일치** → 포장/역포장이 자명. payload는 메타·전송포맷을 모름(C-4 도메인 격리).
- **보상-개시 이벤트 `OrderCancellationRequested(reason)` 신설**로 모든 실패·취소 경로를 이벤트로 닫음(공백 제거).
- **순서 보장 학습 포인트**: 순서는 *파티션 내*에서만 성립. 한 주문 이벤트가 3토픽에 흩어지므로 **토픽 간 전역 순서는 없다** — 그런데 문제가 안 된다. 인과 순서는 **사가 흐름 자체가 강제**하니까(Inventory는 `PaymentCompleted`를 받아야만 차감).

### ⑥ 인바운드 API: 202 + 폴링, 멱등키 전용 테이블 — [ADR-0006](./adr/0006-inbound-api-response-and-idempotency.md)
- **문제**: 동기 HTTP 요청이 비동기 코레오그래피 Saga를 어떻게 응답하나? (기획서가 "즉시 PENDING 응답"과 "확정 응답"을 모순되게 적고 있었음 → 이 ADR이 정정)
- **결정 A**: `202 Accepted {orderId, PENDING}` 즉시 반환 + `GET /orders/{id}` 폴링(read model 재활용). sync-over-async는 HTTP 스레드를 사가 완료(30s)까지 점유 → 고갈 → 기각.
- **결정 B**: 멱등키 전용 테이블 + **order insert와 같은 트랜잭션**.
- **시그니처 인사이트**: **"멱등성도 결국 원자성 문제다."** 키 기록과 주문 생성이 분리되면 "주문 생성·키 미기록 → 재시도 중복" 또는 "키 기록·주문 없음"이 생긴다 — Outbox가 푸는 dual-write 문제와 **동형**.

### ⑦ 헥사고날 레이어: 모듈 vs 패키지 — [ADR-0005](./adr/0005-hexagonal-layer-as-package-vs-module.md)
- 위 §1 원칙 3에서 다룸. 11모듈 → 5모듈, 도메인 순수성을 컴파일러 → ArchUnit으로 강등하되 컨텍스트 격리는 모듈로 사수.

## 3. 핵심: "코레오그래피로 횡단 관심사를 다루면 오케스트레이터를 재발명한다"

면접에서 설계 깊이를 한 문장으로 보여줄 수 있는 **관통 서사**다. 코레오그래피를 택한 대가가 어떻게 곳곳에서 "작은 오케스트레이터"를 다시 자라나게 하는지가 여러 ADR에 일관되게 나타난다.

```
코레오그래피 채택(ADR-0001)
   │  "중앙 상태·중앙 감시자가 없다"는 빚을 짐
   ▼
① 흐름 비가시성  → order_saga_progress read model (ADR-0004)   ← 중앙 상태의 부활(축소판)
② 타임아웃 주체 부재 → Order 데드라인 체커 (ADR-0003)            ← 중앙 감시자의 부활(축소판)
③ 정상확정 vs 타임아웃취소 경합 → 낙관적 락 + 멱등 가드 (ADR-0003) ← 중앙 상태가 없어 Order가 자체 동시성 정리
④ 단계별 타임아웃이 필요해지는 순간 = 워크플로 엔진(Temporal 등)으로 가는 순간 (ADR-0003 §7)
```

핵심 통찰: **"단계별 타임아웃이 필요해지는 순간이 곧 오케스트레이션/워크플로 엔진으로 가야 하는 순간"**이라는 판단. 성숙한 시스템도 단계별 데드라인을 손으로 짜지 않고 durable timer(Temporal·Step Functions·Axon DeadlineManager)로 간다. 그래서 우리 규모에선 **전체 타임아웃 + 폴링 스위퍼가 실무에서도 정석인 build-it-yourself 패턴**이고, 단계별은 의도적으로 미룬다.

> 이 서사가 강한 이유: 하나의 결정(코레오그래피)이 만든 빚을 끝까지 추적하고, 그 빚을 갚는 모든 임시방편이 결국 "포기한 대안(오케스트레이션)"으로 수렴함을 스스로 발견했다는 것. **설계는 단발 선택이 아니라 결과의 연쇄임을 안다**는 신호다.

## 4. 멱등성: 4개 층위로 분산된 횡단 관심사

분산 시스템에서 멱등성을 "한 군데"가 아니라 **흐름의 각 경계마다** 다르게 설계했음을 보여주는 부분.

| 경계 | 위협 | 방어 | 키 | 정책 |
|------|------|------|-----|------|
| 클라이언트 → API | 사용자 중복 클릭/재시도 | 멱등키 전용 테이블 (INSERT-or-catch) | 클라이언트 UUID | PI-1·2·3 |
| Kafka → 컨슈머 | at-least-once 중복 전달 | **Inbox 패턴** | `eventId` | PI-5 |
| 상태 전이 | 중복 이벤트로 같은 전이 재발생 | 전이를 멱등하게(no-op) | 상태 비교 | PS-4 |
| 보상 동작 | 보상 메시지 중복 소비 | 보상도 멱등(이미 보상 시 no-op) | — | PB-2 |

면접 포인트: **"재시도의 전제는 멱등성"**(PT-3)이라는 규칙. 멱등하지 않은 핸들러는 애초에 재시도 대상이 될 수 없게 정책으로 못박았다. 그리고 producer 측 멱등키(PI-1)와 consumer 측 Inbox(PI-5)는 **별개 관심사**임을 ADR-0006/0007 경계로 명확히 분리했다(클라이언트 멱등키는 이벤트 계약에 넣지 않음).

## 5. 의도적으로 안 한 것 (스코프 규율)

면접관이 좋아하는 신호: **무엇을 안 했고, 왜 안 했는지를 안다.** 무지가 아니라 판단으로 뺀 것들.

| 제외 | 이유 |
|------|------|
| 회원/장바구니/배송/정산/쿠폰 | 분산 패턴 학습과 무관 → 핵심 흐름만 남김 |
| 셀러/운영자 페르소나 | **구매자 한 명의 주문 흐름만으로 Saga·멱등·동시성이 100% 커버**됨. 재고는 시드 데이터로 대체 |
| 결제 PG 실연동 | 가짜 PG가 **결정론적 실패 주입**(금액 끝자리 7 → 실패) → 테스트 재현성↑ |
| DB 인스턴스 분리 | 단계 3개 학습 범위엔 오버스펙. 스키마 격리로 충분 |
| Schema Registry(Avro) | 가산적 JSON 규약으로 충분 → 오버스펙 |
| CONFIRMED 후 취소 | 배송·반품 도메인을 끌어들임(Out of scope) → 종료 상태는 불변으로 |

> 핵심: **단일 페르소나 원칙**. "분산 패턴 학습은 구매자 한 명의 주문 흐름만으로 완전히 재현 가능하다"는 판단으로 스코프를 공격적으로 깎았다. 오버셀 동시성조차 *구매자 동시 주문*만으로 100% 재현된다.

## 6. 예상 면접 질문 & 답변 포인트

> 이 프로젝트로 받을 법한 질문과, 어느 문서를 근거로 답할지.

**Q. 왜 오케스트레이션이 아니라 코레오그래피인가? 흐름 가시성은 오케스트레이션이 낫지 않나?**
→ 맞다. 디버깅·가시성만 보면 오케스트레이션이 유리하다. 그런데 (1) 단계 3개·선형이라 중앙 조정자는 오버엔지니어링이고, (2) 학습 목적상 코레오그래피의 고통(흐름 분산·보상 분산·타임아웃 주체 부재)을 먼저 겪는 게 오케스트레이션의 가치를 체득하는 길이라 의도적으로 택했다. 그 빚을 갚는 과정(read model, 데드라인 체커)이 결국 "작은 오케스트레이터"로 수렴함을 §3에서 직접 관찰했다. ([ADR-0001](./adr/0001-saga-orchestration-vs-choreography.md))

**Q. dual-write 문제를 어떻게 푸는가?**
→ Outbox. 상태 변경과 이벤트 적재를 *같은 로컬 트랜잭션*에 쓰고, 별도 릴레이가 비동기 발행. 그리고 같은 문제가 **멱등키 저장에도 동형으로 나타남**을 발견해(키 기록 + 주문 생성을 한 트랜잭션으로) "멱등성도 원자성 문제"로 일반화했다. ([ADR-0004](./adr/0004-schema-separation-outbox-readmodel.md), [ADR-0006](./adr/0006-inbound-api-response-and-idempotency.md))

**Q. Kafka는 at-least-once인데 중복 처리는?**
→ Inbox로 `eventId` 기준 1회만 처리(effectively-once 보정). 전이·보상까지 모두 멱등하게 설계해 재시도가 부작용을 중복시키지 않게 했다. ([policy PI-5](./policy.md), [ADR-0004](./adr/0004-schema-separation-outbox-readmodel.md))

**Q. 동시에 마지막 재고 1개를 두 명이 주문하면?**
→ 기본 어댑터는 원자적 조건부 UPDATE(`WHERE qty >= n`)라 검사·차감이 한 연산 → race 자체가 성립 안 함. 비관/낙관/Redis 어댑터로 갈아끼워 동일 테스트로 "오버셀 0"을 비교 검증한다. ([ADR-0002](./adr/0002-inventory-concurrency.md))

**Q. 비동기 Saga인데 HTTP 응답은 어떻게?**
→ 202 + PENDING 즉시 반환, 확정은 read model 폴링. sync-over-async는 HTTP 스레드를 30s까지 점유해 고갈되므로 기각. ([ADR-0006](./adr/0006-inbound-api-response-and-idempotency.md))

**Q. 타임아웃은 누가 감지하나? 프로세스가 죽으면?**
→ Order의 폴링 스위퍼가 `order_saga_progress.deadline_at`을 스캔. 상태가 DB에 있어 크래시 후 재시작하면 밀린 만료를 그대로 줍는다(인메모리 타이머가 못 가지는 속성). 정상 확정과 타임아웃 취소의 경합은 낙관적 락 + 멱등 가드로 한쪽만 이기게 한다. ([ADR-0003](./adr/0003-order-deadline-checker.md))

**Q. 이벤트 순서는 보장되나?**
→ 파티션 키 `orderId`로 *파티션 내* 순서만 보장. 토픽 3개에 흩어지므로 전역 순서는 없지만, 인과 순서는 사가 흐름이 강제하므로(차감은 결제 완료 이후) 전역 순서에 의존하지 않는다. ([ADR-0007](./adr/0007-integration-event-contract.md))
