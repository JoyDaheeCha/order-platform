# policy — order-platform 도메인 정책서 (v1.0)

> **목적**: 주문 도메인의 **비즈니스 규칙과 불변식(invariant)**을 단일 진실 공급원(Single Source of Truth)으로 명문화한다.
> 구현(코드·테스트)은 이 문서의 규칙을 따른다. 페르소나·스코프는 [`product-spec.md`](./product-spec.md) 참조.
>
> 각 정책은 **규칙(Rule) · 근거(Why) · 트레이드오프** 형식으로 적는다. "왜 이렇게 정했나"를 남겨야 나중에 바꿀 때 판단 근거가 된다.

---

## 0. 표기 규약

- 정책 ID: `P{영역}-{번호}` (예: `PO-1` = 주문 정책 1번). 코드/테스트/PR에서 이 ID로 참조한다.
- **MUST / MUST NOT** = 강제 불변식 (위반 시 도메인 예외). **SHOULD** = 기본 정책 (예외 가능).
- 통화는 **KRW, 정수(원)** 단위. 소수·부동소수 금지.

## 1. 이벤트 · 커맨드 · Aggregate 맵 (정책 앵커)

> 각 정책이 "어디서 발동되는가"를 고정하기 위한 흐름 지도. (이벤트 스토밍의 구조를 압축)

| Aggregate | 받는 커맨드 (Command) | 내는 이벤트 (Event) |
|-----------|----------------------|---------------------|
| **Order** | `PlaceOrder`, `CancelOrder` | `OrderPlaced`, `OrderConfirmed`, `OrderCancellationRequested`, `OrderCancelled` |
| **Payment** | `ProcessPayment`, `RefundPayment` | `PaymentCompleted`, `PaymentFailed`, `PaymentRefunded` |
| **Inventory** | `DeductStock`, `RestoreStock` | `StockDeducted`, `StockShortage`, `StockRestored` |

정상 흐름:
```
PlaceOrder → OrderPlaced ─▶ ProcessPayment → PaymentCompleted ─▶ DeductStock → StockDeducted ─▶ OrderConfirmed
```
보상 흐름(역순):
```
[하류 실패]   StockShortage ─▶ RefundPayment → PaymentRefunded ─▶ OrderCancelled
[Order 개시]  OrderCancellationRequested(reason) ─▶ 진행분 보상(RefundPayment/RestoreStock) ─▶ OrderCancelled
```
> `OrderCancellationRequested`는 **Order가 먼저 취소를 선언**하는 경로(사용자 취소 PC-3 / 타임아웃 PT-1)의 보상-개시 이벤트다(`reason` = USER_CANCEL | TIMEOUT). 하류 실패(E1·E2)는 실패 이벤트가 직접 보상을 트리거하므로 이 이벤트를 쓰지 않는다. 계약 상세는 [ADR-0007](./adr/0007-integration-event-contract.md).

> Saga는 **코레오그래피**로 구현한다([ADR-0001](./adr/0001-saga-orchestration-vs-choreography.md), Accepted). 중앙 오케스트레이터 없이 각 컨텍스트가 이벤트를 구독해 자율 반응한다. 단, 커맨드(`ProcessPayment` 등)는 *외부 메시지가 아니라* 해당 이벤트 수신 시 컨텍스트 내부에서 실행되는 동작으로 본다. (외부 커맨드는 구매자발 `PlaceOrder`·`CancelOrder`뿐) 이 정책서의 규칙은 구현 방식과 무관하게 지켜진다.

## 2. 주문 생성 정책 (PO)

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PO-1** | 주문은 1개 이상의 주문 라인으로 구성된다. 라인 0개 주문은 MUST NOT 생성. | 빈 주문은 의미 없음. |
| **PO-2** | 각 주문 라인의 수량은 **1 이상 99 이하 정수**. | 상한 99는 학습용 임의값(대량주문 도메인 복잡도 제거). 음수·0·소수 금지. |
| **PO-3** | 동일 상품(productId)이 한 주문에 중복 라인으로 오면 MUST 거부한다(병합하지 않음). | 병합 로직은 도메인 본질과 무관 → 클라이언트가 합쳐 보내도록 강제해 단순화. |
| **PO-4** | 주문 총액 = Σ(주문 시점 단가 × 수량). 단가는 **주문 생성 시점에 스냅샷**되어 주문에 고정된다. | 이후 상품 가격이 바뀌어도 주문 금액 불변(가격 정합성). |
| **PO-5** | 주문 생성 시 상품 존재·가격은 시드 데이터 기준으로 검증한다. 없는 상품 MUST 거부. | 카탈로그는 Out of scope이므로 시드가 진실. |

## 3. 멱등성 정책 (PI) — 분산 학습 핵심

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PI-1** | `PlaceOrder` 요청은 클라이언트가 생성한 **멱등키(UUID)**를 MUST 포함한다. | 네트워크 재시도로 인한 중복 주문 방지(생산자 측 멱등성). |
| **PI-2** | 멱등키는 **전역 유니크**, TTL **24시간**. 동일 키 재요청 시 새 주문을 만들지 않고 **기존 주문 결과를 그대로 반환**한다. | 24h는 사용자 재시도 현실 범위. TTL 후 키 재사용은 새 주문으로 간주. |
| **PI-3** | 동일 멱등키 + **다른 페이로드**로 오면 MUST `409 Conflict`로 거부한다. | 같은 키로 다른 주문을 시도 = 클라이언트 버그. 조용히 무시하면 안 됨. |
| **PI-4** | 모든 도메인 이벤트는 `eventId`(UUID), `occurredAt`, `orderId`(상관관계 키)를 MUST 포함한다. | 소비자 멱등성·추적·Saga 상관관계의 전제. |
| **PI-5** | 모든 이벤트 소비자는 **Inbox 패턴**으로 `eventId` 기준 중복 소비를 1회만 처리한다. 처리 기록 보관 TTL **7일**. | 소비자 측 멱등성. Kafka의 at-least-once 전달을 effectively-once로 보정. |
| **PI-6** | 이벤트 발행은 **Outbox 패턴**으로 DB 트랜잭션과 원자적으로 기록 후 비동기 발행한다. | 비즈니스 상태 변경과 이벤트 발행의 원자성(dual-write 문제 해결). |

## 4. 주문 상태 전이 정책 (PS) — 불변식

상태: `PENDING` → `PAID` → `CONFIRMED` / `CANCELLED`

```
  [PENDING] ──결제완료──▶ [PAID] ──재고차감──▶ [CONFIRMED] (종료)
      │                     │
      │                     └─재고부족──▶ [CANCELLED] (환불 보상 후, 종료)
      └─결제실패/사용자취소──────────────▶ [CANCELLED] (종료)
```

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PS-1** | 위 다이어그램에 **명시된 전이만 MUST 허용**한다. 그 외 모든 전이는 도메인 예외. | 불법 전이(예: `PENDING`→`CONFIRMED` 건너뛰기) 차단. |
| **PS-2** | `CONFIRMED`, `CANCELLED`는 **종료 상태(terminal)**. 종료 상태에서의 어떤 전이도 MUST NOT 허용. | 멱등성·정합성. 종료된 주문 재처리 방지. |
| **PS-3** | 상태 전이는 **도메인 이벤트 수신에 의해서만** 일어난다. 외부에서 상태를 임의 설정 MUST NOT. | 상태와 이벤트의 1:1 정합성 유지. |
| **PS-4** | 각 전이는 멱등하다 — 이미 그 상태면 재전이는 **무시(no-op)**하고 예외를 던지지 않는다. | 이벤트 중복 소비 시(PI-5와 연계) 안전. |

## 5. 결제 정책 (PP)

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PP-1** | 결제 금액은 주문 총액(PO-4)과 **정확히 일치**해야 한다. 불일치 시 결제 MUST 거부 → 보상. | 금액 변조·계산오류 방어. |
| **PP-2** | 결제는 **1회·전액**. 부분 결제·분할 결제 MUST NOT. 실패 시 재결제 없이 주문 취소. | 학습 단순화. 재결제 흐름은 보상 학습과 무관. |
| **PP-3** | 가짜 PG는 **결정론적으로** 실패를 주입한다: **결제 금액의 끝자리가 `7`이면 실패**(`PaymentFailed`), 그 외 성공. | 확률 기반보다 결정론이 테스트 재현성↑. 특정 금액으로 실패 시나리오를 항상 재현 가능. |
| **PP-4** | 환불(`RefundPayment`)은 **전액**만 지원. 부분 환불 MUST NOT. | 부분취소가 Out of scope이므로 부분환불도 불필요. |

## 6. 재고 정책 (PV)

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PV-1** | 가용재고 = **물리 재고 수량**. 별도 "예약(reservation)" 개념을 두지 않는다. | 단순화. 예약 모델은 향후 확장 학습 과제. |
| **PV-2** | 재고 차감은 **`PaymentCompleted` 수신 후**(`PAID` 단계) 시도한다. | 결제 확정 후 차감 → 결제 후 재고부족(E2)이 *진짜 보상*을 만든다. |
| **PV-3** | 차감은 주문 단위 **all-or-nothing**. 한 라인이라도 부족하면 전체 차감을 하지 않고 `StockShortage`. | 부분 주문 성립은 Out of scope. 전부 되거나 전부 보상. |
| **PV-4** | 동시 차감 시 **오버셀 MUST NOT**(가용재고 미만으로 내려갈 수 없다). | 동시성 학습의 핵심 불변식. *구체 기법*(비관적 락 / 낙관적 락 / 원자적 UPDATE / Redis 4종을 포트 어댑터로 비교 구현, 기본 B)은 [ADR-0002](./adr/0002-inventory-concurrency.md) Accepted. |
| **PV-5** | 보상 복구(`RestoreStock`)는 **차감했던 수량을 정확히 원복**한다. 차감 전 실패면 복구 불필요(no-op). | 보상의 정확성·멱등성(PI-5와 연계). |

## 7. 취소 / 환불 정책 (PC)

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PC-1** | 사용자 취소(`CancelOrder`)는 `PENDING`, `PAID` 상태에서만 MUST 허용. `CONFIRMED` 후에는 MUST NOT. | **트레이드오프 명시**: 현실 커머스는 확정 후 취소가 흔하나, 그건 배송·반품 도메인을 끌어들임(Out of scope). 학습 범위에서 종료 상태는 불변으로 둔다. |
| **PC-2** | `PENDING`에서 취소 → 결제 전이므로 **단순 취소**(보상 없음), `OrderCancelled`. | 보상 대상이 없음. |
| **PC-3** | `PAID`에서 취소 → 결제 완료분 **전액 환불 보상** 후 `OrderCancelled`. 재고는 아직 미차감이므로 복구 없음(PV-2). | 사용자 트리거 보상의 대표 케이스(E6). |
| **PC-4** | 취소 요청이 **재고 차감 진행과 경합**하면(`PAID`↔`CONFIRMED` 전이 중) Saga 상태로 **직렬화**한다 — 차감이 끝나 `CONFIRMED`되면 PC-1에 의해 취소는 거부된다. | 경쟁 조건을 정책으로 명확화. 취소와 확정은 동시에 성립할 수 없다. |

## 8. 보상 트랜잭션 정책 (PB) — 분산 학습 핵심

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PB-1** | 보상은 **수행된 정방향 단계의 역순**으로 실행한다(결제완료→재고차감 진행분만 역순 보상). | Saga 보상의 기본 원칙. 수행 안 된 단계는 보상하지 않는다. |
| **PB-2** | 모든 보상 동작(`RefundPayment`, `RestoreStock`)은 **멱등**해야 한다 — 이미 보상됐으면 재실행은 no-op. | 보상 메시지 중복 소비 대비(PI-5). |
| **PB-3** | 보상은 **최종적으로 반드시 완료**된다(eventual consistency). 일시적 실패는 재시도, 한계 초과 시 **DLQ + 경보(학습용 로그)**로 보내되 포기하지 않는다. | 보상 누락 = 데이터 불일치(돈/재고). 절대 유실 금지. |
| **PB-4** | 비즈니스 실패(결제 실패 PP-3, 재고 부족 PV-3)는 **재시도하지 않고 즉시 보상**으로 전환한다. | 재시도해도 결과가 같음. 일시적 오류와 구분(PT-2). |

## 9. 타임아웃 / 재시도 / DLQ 정책 (PT)

| ID | 규칙 | 근거 / 트레이드오프 |
|----|------|--------------------|
| **PT-1** | Saga의 각 단계는 **타임아웃**을 가진다. 응답 무응답이 한계를 넘으면 **실패로 간주**하고 보상으로 전환한다. | 무한 대기 방지. 입도·메커니즘·수치는 [ADR-0003](./adr/0003-order-deadline-checker.md)(전체 사가 타임아웃 + DB 스위퍼, 기본 30s). |
| **PT-2** | 소비자는 **일시적 오류**(네트워크·DB 순단)만 **지수 백오프 재시도**한다. 최대 횟수 초과 시 DLQ. | 일시 오류는 재시도로 회복 가능, 영구 오류는 재시도 무의미. |
| **PT-3** | 재시도의 전제는 **멱등성**이다(PI-5, PS-4, PB-2). 멱등하지 않은 핸들러는 재시도 대상이 될 수 없다. | 재시도가 부작용을 중복시키면 안 됨. |
| **PT-4** | DLQ로 간 메시지는 **유실로 간주하지 않으며**, 수동/배치 재처리 경로를 둔다(학습용 최소 구현). | at-least-once 보장의 마지막 안전망. |

## 10. 구현 결정으로 미루는 것 (정책 아님 → ADR)

> 아래는 "규칙"이 아니라 "구현 방식"이라 정책서가 아닌 ADR에서 결정한다. 어느 쪽을 택하든 위 정책은 동일하게 지켜진다.

- **ADR-0001** ✅ Accepted: Saga — **코레오그래피** 채택 (오케스트레이션은 후속 학습)
- **ADR-0002** ✅ Accepted: 재고 동시성 기법 — 비관적 락 / 낙관적 락 / 원자적 UPDATE / Redis 4종을 `StockDeducer` 포트 어댑터로 비교 구현(기본 B), 공통 동시성 테스트로 검증 (PV-4 충족)
- **ADR-0003** ✅ Accepted: Order 데드라인 체커 — 전체 사가 타임아웃 + DB 스위퍼(폴링) 채택, 재시도/DLQ(비즈니스 실패 ↔ 인프라 오류 분리) (PT 충족)
- **ADR-0004** ✅ Accepted: 모듈별 스키마 분리(단일 인스턴스, 컨텍스트별 DataSource), Outbox/Inbox 테이블, 주문 상태 read model (PI-5·PI-6 충족)
