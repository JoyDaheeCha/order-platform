# order-platform 설계 설명

### 분산 트랜잭션을 단일 DB 트랜잭션 없이 Saga로 맞추기

<br>

발표자: 차다희 (Joy Dahee Cha)

---

## 한 문장으로

> 주문 · 결제 · 재고 **세 컨텍스트에 걸친 분산 트랜잭션**을,
> 단일 DB 트랜잭션 없이 **Saga와 이벤트 기반으로 정합성 있게** 맞추는 주문 도메인.

**핵심 결정 3가지** (나머지는 이 셋의 파생)

1. Saga는 **코레오그래피** (중앙 조정자 없음)
2. 데이터는 **컨텍스트별 스키마 분리 + Outbox/Inbox**
3. **모듈러 모노리스** — 단일 배포, 경계만 MSA 수준 강제

> 모든 결정은 *트레이드오프와 함께* ADR 7건으로 기록.

---

## 왜 어려운 문제인가

주문 하나가 성립하려면 **세 가지가 다 맞아야** 한다.

```
  주문 생성   →   결제   →   재고 차감
  (order)      (payment)   (inventory)
```

문제: 셋이 **서로 다른 컨텍스트**에 있다.
→ 옛날처럼 "하나의 DB 트랜잭션으로 묶기"가 **불가능**.

**그래서 Saga + 보상 트랜잭션이 불가피하다.**
"셋 다 되거나, 한 일을 거꾸로 되돌리거나" — 이 보장이 프로젝트의 전부.

---

## 결정 ① Saga — 코레오그래피

중앙 오케스트레이터 없이, 각 컨텍스트가 **이벤트를 구독해 자율 반응**.

```
OrderPlaced ─▶ [Payment] ─ PaymentCompleted ─▶ [Inventory] ─ StockDeducted ─▶ [Order] confirm
                  │                                  │
            PaymentFailed ─▶ cancel           StockShortage ─▶ refund ─▶ cancel
```

**왜 오케스트레이션이 아니라?**
- 단계가 3개·선형 → 중앙 조정자는 **오버엔지니어링**
- (학습 목적) 코레오그래피의 *고통*을 먼저 겪는 게 오케스트레이션의 가치를 체득하는 길

> **트레이드오프**: 결합 최소·인프라 단순 ↔ 흐름 비가시성·타임아웃 감지 주체 부재

---

## 주문 상태 머신 (이 그림 하나로 다 설명된다)

```
   [PENDING] ──결제완료──▶ [PAID] ──재고차감──▶ [CONFIRMED]  ✅ 성공 종료
       │                     │
       │                     └─ 재고부족 ─▶ [CANCELLED]  (결제 환불 보상)
       │
       └─ 결제실패 / 사용자취소 ─────────▶ [CANCELLED]
```

- **Happy path**: 생성 → 결제 → 차감 → 확정
- **보상**: 결제는 됐는데 재고 부족 → *이미 한 일(결제)만* 역순 환불
- **종료 상태**(CONFIRMED/CANCELLED)는 불변 — 재처리 차단

---

## 결정 ② 데이터 — 세 결정이 맞물려 도출된다

```
스키마를 분리하면
   → 컨텍스트 간 JOIN/FK/단일 트랜잭션이 불가능
   → "어디까지 진행됐나"를 JOIN으로 못 봄    →  read model 필요
   → "상태 변경 + 이벤트 발행"을 한 Tx로 못 묶음(dual-write) → Outbox 필요
```

| 결정 | 내용 |
|------|------|
| 스키마 분리 | 컨텍스트별 스키마 (단일 MySQL) + **컨텍스트별 DataSource** |
| Outbox | 상태 변경과 *같은 트랜잭션*에 이벤트 적재 → 별도 릴레이가 발행 |
| Inbox | `eventId` 기준 1회만 처리 (at-least-once → effectively-once) |

> cross-schema 접근이 **물리적으로 불가능** → 나중에 서비스 분리가 평이.

---

## Outbox 패턴 — dual-write 문제와 해법

**문제**: "DB 상태 변경"과 "Kafka 발행"은 다른 시스템 → 한 트랜잭션에 못 묶임.

```
save(order); kafka.send(evt);   ← 사이에서 죽으면? 주문은 있는데 이벤트 유실 ❌
```

**해법**: Kafka로 직접 보내지 말고, **같은 트랜잭션에서 outbox 테이블에 INSERT**.

```
┌──── 한 로컬 트랜잭션 (order_schema) ────┐
│  orders     INSERT   (비즈니스 상태)     │  ← 원자적으로
│  order_outbox INSERT (보낼 이벤트)       │     같이 커밋
└──────────────────────────────────────────┘
              │ (비동기) 폴링 릴레이가 published_at IS NULL 행을 발행
              ▼
          Kafka (order.events)
```

> **"이벤트 발행"을 "DB 쓰기"로 바꿔** 원자성을 회복한다 (정책 PI-6).

---

## Outbox 시퀀스 & 정합성

```
[application]  @Transactional { save(order) + outbox.append(evt) }   ← 원자적 커밋
                                    │
[infrastructure] 도메인이벤트 → 통합이벤트 변환(C-4) → EventEnvelope 포장 → outbox INSERT
                                    │
[relay] @Scheduled: published_at IS NULL 조회 → Kafka 발행 → published_at 기록
                                    │
[consumer] Inbox(eventId)로 중복 제거 → 1회만 처리
```

| 보장 | 메커니즘 |
|------|----------|
| 원자성 | 같은 DataSource + `@Transactional` 하나 |
| at-least-once | 발행 전 크래시 → 행이 남아 다음 폴링이 줍는다 |
| **중복 발행** | "발행 성공 → 기록 직전 크래시"면 재발행 → **소비자 Inbox가 제거** |
| 순서 | 파티션 키 = `orderId` → 같은 파티션 |

> **Outbox는 "최소 한 번"만 보장. exactly-once는 Inbox와 짝을 이뤄야 완성된다.**

## 결정 ③ 모듈러 모노리스 — 경계 강제의 3단계

"경계를 지키자"가 아니라, **무엇이 무엇을 어떻게 막는지**를 결정.

| 강제 대상 | 메커니즘 | 강제력 |
|-----------|----------|--------|
| 컨텍스트 간 코드 의존 | **모듈 경계** | 컴파일타임 |
| 레이어 의존 방향 | **ArchUnit 테스트** | 빌드(테스트) |
| 컨텍스트 간 데이터 결합 | **컨텍스트별 DataSource** | 물리적 불가능 |

> 헥사고날 레이어는 모듈이 아니라 **패키지**로 (11모듈 → 5모듈).
> "DDD의 본질은 컨텍스트 경계지 레이어-모듈 분리가 아니다"는 판단.

---

## 깊이의 한 방 🎯

> ### "코레오그래피로 횡단 관심사를 다루면, 결국 오케스트레이터를 재발명하게 된다"

```
코레오그래피 채택 → "중앙 상태·중앙 감시자가 없다"는 빚
   │
   ├─ 흐름 비가시성     → order_saga_progress read model   (중앙 상태의 부활)
   ├─ 타임아웃 주체 부재 → Order 데드라인 체커               (중앙 감시자의 부활)
   └─ 정상확정 vs 타임아웃 경합 → 낙관적 락 + 멱등 가드       (자체 동시성 정리)
```

**통찰**: "단계별 타임아웃이 필요해지는 순간 = 워크플로 엔진(Temporal 등)으로 가야 하는 순간."
→ 그래서 우리 규모에선 **전체 타임아웃 + 폴링 스위퍼**가 정석. 단계별은 *의도적으로 미룸*.

---

## 멱등성 — 한 군데가 아니라 4개 층위

| 경계 | 위협 | 방어 |
|------|------|------|
| 클라이언트 → API | 중복 클릭/재시도 | 멱등키 전용 테이블 (INSERT-or-catch) |
| Kafka → 컨슈머 | at-least-once 중복 | **Inbox 패턴** (`eventId`) |
| 상태 전이 | 중복 이벤트 | 전이를 멱등하게 (no-op) |
| 보상 동작 | 보상 메시지 중복 | 보상도 멱등 |

> **"재시도의 전제는 멱등성"** — 멱등하지 않은 핸들러는 애초에 재시도 대상이 못 됨.
> 그리고 **"멱등성도 결국 원자성 문제"** — 멱등키 저장이 Outbox dual-write와 동형.

---

## 재고 동시성 — 포트 + 4어댑터 비교

동일 상품 동시 주문 → read-modify-write 경쟁 → **오버셀**.

```
application:   interface StockDeducer            ← 아웃바운드 포트
infrastructure: 비관적락 · 낙관적락 · 원자적UPDATE(기본) · Redis  ← 어댑터 4개
```

- **기본 = 원자적 조건부 UPDATE** (`WHERE qty >= n`)
  → 검사·차감이 한 연산이라 **"충돌" 개념 자체가 없음**
- 도메인·정책 **무변경**으로 4기법 교체 → 공통 테스트로 "오버셀 0" 비교 검증

> 헥사고날이 장식이 아니라 *실제로 작동함*을 증명하는 부분.

---

## 의도적으로 안 한 것 (스코프 규율)

무지가 아니라 **판단**으로 뺀 것 — 오버엔지니어링 방어.

| 제외 | 이유 |
|------|------|
| 회원/장바구니/배송/정산 | 분산 패턴 학습과 무관 |
| 셀러/운영자 페르소나 | **구매자 한 명 주문 흐름만으로 Saga·멱등·동시성 100% 커버** |
| 결제 PG 실연동 | 가짜 PG가 결정론적 실패 주입 → 테스트 재현성↑ |
| DB 인스턴스 분리 | 학습 범위엔 오버스펙 (스키마 격리로 충분) |
| 단계별 타임아웃 | 필요해지는 순간 = 워크플로 엔진으로 가는 순간 |

---

## 예상 질문 & 대응

**Q. 왜 오케스트레이션 안 했나?**
→ 가시성은 그게 낫다. 단 3단계 선형엔 과하고, 빚이 결국 오케스트레이터로 수렴함을 직접 관찰.

**Q. 모노리스인데 스키마까지 나눌 필요?**
→ DataSource로 cross-schema를 *물리적으로* 차단 → 서비스 분리 시 DataSource만 떼면 됨.

**Q. Kafka at-least-once인데 중복은?**
→ Inbox로 1회 처리. 전이·보상까지 전부 멱등이라 재시도가 부작용을 중복 안 시킴.

**Q. 오버엔지니어링 아닌가?**
→ 그래서 회원·배송·셀러를 다 뺐다. 핵심 분산 패턴만 남김.

---

## 한계 & 다음 단계 (스스로 인정하는 부분)

- **검출 지연 = 스위퍼 주기** — 정밀 SLA엔 부적합 (주문 타임아웃엔 충분)
- **read model 최종 일관성** — 조회 시점 stale 가능
- **스위퍼 다중 인스턴스** — 현재 단일 배포라 무관, 분리 시 `SKIP LOCKED` 필요
- **다음 학습**: 같은 도메인을 **오케스트레이션/워크플로 엔진으로 재구현**해 트레이드오프 비교

# 정리

**분산 정합성을, 단일 트랜잭션 없이 Saga로.**

택한 것 = 코레오그래피 · 스키마 분리 · 모듈러 모노리스
포기한 것 = 흐름 가시성 · 중앙 상태 → read model·데드라인 체커로 보완
미룬 것 = 오케스트레이션 · 단계별 타임아웃 (필요해지는 순간을 알기에)
