# product-spec — order-platform 기획서 (v1.0)

> **목적**: 이 문서는 제품 출시용 PRD가 아니라 **이벤트 스토밍의 입력**이다.
> 주문 도메인을 이벤트로 분해할 수 있을 만큼만 정의하고, 학습 목표(Kafka/EDA/DDD·분산패턴)에서 멀어지는 기능은 과감히 Out of scope로 둔다.

---

## 1. 한 줄 정의

디자이너 브랜드 오픈마켓에서 **구매자가 상품을 주문하고, 결제와 재고가 분산 트랜잭션(Saga)으로 정합성을 맞추는** B2C 주문 도메인.

---

## 2. 페르소나 & 액터

| 페르소나 | 역할 | 등급 | 핵심 액션 | 학습 연결 |
|----------|------|------|-----------|-----------|
| **구매자 (Buyer)** | B2C 소비자 | 1급 (유일) | 주문 생성, 주문 취소 | Saga 시작점, 멱등성(중복 주문 방지) |

> **구매자 단일 페르소나** 원칙. 분산 패턴 학습(Saga·멱등성·Inbox·동시성)은 구매자 한 명의 주문 흐름만으로 100% 커버된다.
>
> **재고는 셀러 없이 시드 데이터로 미리 적재**한다. 셀러가 빠져도 inventory는 독립 컨텍스트로 살아있고, 오버셀 동시성은 *구매자 동시 주문*만으로 완벽히 재현된다.
>
> 셀러/운영자는 "확장 학습 과제"로 §3 Out of scope 참고.

---

## 3. 스코프

### In scope
- 구매자의 **주문 생성 → 결제 → 재고 차감** Saga (happy path)
- 실패 시 **보상 트랜잭션** (결제 실패, 재고 부족, 사용자 취소)
- **멱등성**: 중복 주문 요청, 중복 이벤트 소비 방어
- **재고 동시성 제어**: 동일 상품 동시 주문 시 오버셀 방지
- 모듈 간 통신은 **Kafka 이벤트 only** (Outbox/Inbox 포함)

### Out of scope
- 회원/인증, 장바구니, 상품 카탈로그 검색, 배송 추적, 정산
- 결제 PG 실연동 (가짜 PG 어댑터로 성공/실패만 시뮬레이션)
- 쿠폰/프로모션/포인트, 부분 취소·부분 환불, 교환/반품
- **셀러 페르소나** (상품/재고 등록·조정) → 재고는 시드 데이터로 대체
- **운영자 페르소나** (강제 취소·환불 승인 등 어드민 액션 전반)

### 확장 학습 과제 (핵심 완료 후 선택)
- **셀러 추가** → 재고 쓰기 양쪽(등록+차감)에서 오는 일관성, inventory 컨텍스트 심화
- **운영자 추가** → 사람이 트리거하는 보상(외부 Command→Saga), 권한/경계 학습

---

## 4. 핵심 유저 플로우 (Happy Path)

> 이 시퀀스가 이벤트 스토밍의 1차 이벤트 원천이 된다.

1. 구매자가 상품·수량을 골라 **주문 생성** 요청 (멱등키 동반)
2. 주문이 `PENDING` 상태로 생성되고 → **`OrderPlaced`** 이벤트 발행
3. 결제 컨텍스트가 구독 → 결제 시도 → 성공 시 **`PaymentCompleted`**
4. 재고 컨텍스트가 구독 → 재고 차감 → 성공 시 **`StockDeducted`**
5. 주문이 `CONFIRMED`로 전이 → **`OrderConfirmed`** 발행
6. 구매자에게 주문 확정 응답

---

## 5. 예외 / 엣지 시나리오 (보상 트랜잭션 학습 포인트)

| # | 시나리오 | 처리 | 학습 포인트 |
|---|----------|------|-------------|
| E1 | 결제 실패 | 주문 → `CANCELLED`, **`OrderCancelled`** 발행 (재고 차감 전이라 복구 불필요) | Saga 보상 시작점 |
| E2 | 결제 성공 후 **재고 부족** | 결제 환불 보상(`PaymentRefunded`) → 주문 `CANCELLED` | 결제 차감 후 보상 = 진짜 보상 트랜잭션 |
| E3 | 중복 주문 요청 (같은 멱등키) | 새 주문 생성 안 함, 기존 결과 반환 | 멱등성 (생산자 측) |
| E4 | 동일 이벤트 중복 소비 | Inbox로 1회만 처리 | 멱등성 (소비자 측) |
| E5 | 동시 주문 → 재고 경합 | 락/원자적 차감으로 오버셀 방지 | 재고 동시성 제어 |
| E6 | 구매자가 `CONFIRMED` 전 취소 | 진행 중 Saga 보상 트리거 (결제 완료분 환불 포함) | **사용자 트리거 보상의 대표 케이스** |

## 6. 주문(Order) 상태 정의

> Aggregate 상태 전이. 각 전이가 도메인 이벤트와 1:1로 매핑된다.

```
                 결제실패(E1)/취소(E6)
        ┌──────────────────────────────────┐
        ▼                                   │
  [PENDING] ──결제완료──> [PAID] ──재고차감──> [CONFIRMED]
        │                   │
        │                   └─재고부족(E2)─> [CANCELLED] (결제 환불 보상)
        │
        └─결제실패(E1)─────────────────────> [CANCELLED]
```

| 상태 | 의미 | 진입 이벤트 |
|------|------|-------------|
| `PENDING` | 주문 생성, 결제 대기 | `OrderPlaced` |
| `PAID` | 결제 완료, 재고 차감 대기 | `PaymentCompleted` |
| `CONFIRMED` | 재고까지 확정, 주문 성립 | `OrderConfirmed` |
| `CANCELLED` | 실패/취소로 종료 (보상 완료) | `OrderCancelled` |
